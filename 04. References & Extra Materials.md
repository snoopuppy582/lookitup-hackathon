Use this as the main problem example.

### Claim

> "Kamchatka snow reached the height of a 10-story apartment building."

### Source Pack

`Breaking News + Fact-Check Pack`

### Evidence Cards

| Source | Evidence Card Text | URL |
| --- | --- | --- |
| OhmyNews Fact Check, Jan. 22, 2026 | Korean media reported AI-manipulated Kamchatka snow visuals as real; the fact-check marked the claim false. | https://www.ohmynews.com/NWS_Web/OhmyFact/at_pg.aspx?CNTN_CD=A0003200915 |
| OhmyNews follow-up, Jan. 23, 2026 | KBS and MBC later deleted related AI snow videos after verification problems were raised. | https://www.ohmynews.com/NWS_Web/View/at_pg.aspx?CNTN_CD=A0003201244 |

### Stage Wording

Use:

> AI-generated visuals from Kamchatka crossed platforms, languages, and newsrooms before evidence caught up.

## N. Q&A Defense

| Question | Answer |
| --- | --- |
| How is this different from a search engine? | The search boundary is selected before the deadline. LookItUp searches inside that trusted pack and returns Evidence Cards. |
| Is this automatic fact-checking? | It is pre-publication evidence search. The journalist decides. |
| Why start with local data? | The pitch runs from local data. URL/RSS ingestion comes after the core loop works. |
| Does P0 need an API key? | No. P0 runs from local trusted-source data. |
| Does P0 need a database? | No production database. A local corpus or lightweight local index is enough for the demo. |
| Is Kamchatka an image-verification product? | Kamchatka is the problem case. The P0 product is trusted-source evidence search before publication. |
| What is the technical core? | Controlled retrieval: source packs define the search boundary, and Evidence Cards make evidence visible. |
