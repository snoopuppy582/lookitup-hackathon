# 1. 해커톤 운영 안내 정리

## 1) 행사 목적

이번 해커톤은 **AI와 미디어/저널리즘의 변화**를 주제로, 실제 기자들의 업무에서 나오는 문제점, 즉 **pain point**를 듣고 48시간 동안 기술 프로토타입으로 해결안을 만드는 행사입니다.

초반 진행자는 학생들에게 단순히 “멋진 아이디어”가 아니라, 기자들이 실제 업무에서 겪는 문제를 듣고 **현실적 문제 정의 → 기술적 해결 → 발표**까지 연결하라고 설명했습니다. 출처: `epitech part1`, 00:00–02:26

---

## 2) 장소와 진행 방식

| 항목 | 내용 |
| --- | --- |
| 킥오프 장소 | 현재 방은 킥오프/발표용 |
| 실제 작업 공간 | 5층, KD4 / International Room로 언급됨 |
| 질문/도움 | 담당자들이 현장 또는 메시지/메일로 지원 |
| 후속 피드백 | 오늘과 내일 optional follow-up 가능 |
| 최종 발표 | 다음 날 오후 4시 |
| 수상 발표 | 다음 날 오후 6시 |

출처: `epitech part1`, 00:35–03:05

---

## 3) 프로젝트 필수 조건

핵심 조건은 다음입니다.

| 조건 | 설명 |
| --- | --- |
| 실제 문제여야 함 | 현실에 존재하는 문제를 해결해야 함 |
| 수치 근거 필요 | 온라인에서 찾은 통계/수치로 문제 규모를 증명해야 함 |
| 글로벌 문제성 | 국제 학생들은 이 문제가 특정 개인이 아니라 더 넓은 집단/세계적 문제임을 보여야 함 |
| 기술 프로젝트 필요 | 단순 아이디어가 아니라 작동하는 프로토타입 필요 |
| 핵심 기능 우선 | 로그인, DB, 전체 웹앱 완성보다 **core feature 작동**이 중요 |
| 구현 형태 | 웹사이트, 앱, IoT 등 가능 |
| AI/LLM 사용 허용 | 단, 발표 때 AI를 어디에 얼마나 썼는지 설명해야 함 |

진행자는 “완성된 풀 웹사이트나 앱을 요구하는 것이 아니다. 핵심 기능이 작동해야 한다”고 강조했습니다. 출처: `epitech part1`, 03:05–04:57

---

## 4) 발표 요구사항

최종 발표는 **5분 elevator pitch** 형식입니다.

발표에서 포함해야 할 것:

| 발표 요소 | 내용 |
| --- | --- |
| 문제 정의 | 어떤 pain point/problematic을 해결하는지 |
| 해결 방식 | 무엇을 만들었는지 |
| 데모 | 무대에서 작동하는 핵심 기능 시연 |
| AI 사용 설명 | AI를 100% 썼는지, 시각자료만 썼는지, 코딩에 썼는지 등 |
| 소스코드 링크 | 프로젝트 검증을 위한 코드 링크 |
| 시각 자료 | 포스터 또는 프로젝트를 설명하는 간단한 visual |
| 타임라인 | 프로젝트 진행/향후 계획을 보여주는 timeline |

출처: `epitech part1`, 04:23–05:28

---

## 5) 수상 부문

수상 부문은 세 가지로 설명되었습니다.

| 상 | 의미 |
| --- | --- |
| Best Project | 심사위원들이 전체적으로 가장 좋다고 판단한 프로젝트 |
| Best Technical Project | 기술적으로 가장 뛰어나거나 완성도가 높은 프로젝트 |
| Best Journalistic Project | 기자/저널리즘 문제에 가장 잘 응답한 프로젝트 |

추가로, 후반부에서 **국제 혼합 팀이 수상에서 선호될 수 있다**고 언급되었습니다. 최소 두 국적 이상이 섞인 팀이 유리하다는 취지였습니다. 팀 규모는 녹취상 “3명”으로 들립니다. 출처: `epitech part1`, 05:28–07:41; `epitech part2`, 25:50–26:40

# 3. 기자들의 일상 업무

## 1) 뉴스 에이전시 기자의 일상

참석자 2는 기자라고 하면 현장 취재와 인터뷰를 떠올리지만, 본인의 일은 대부분 **컴퓨터 앞에서 7시간 동안 빠르게 기사 처리**하는 것이라고 설명했습니다.

업무 흐름:

1. 전 세계 특파원들이 프랑스어/영어로 기사를 보냄.
2. 스페인어 부서에 모든 기사 리스트가 들어옴.
3. 스페인어권 고객에게 중요한 기사인지 판단.
4. 기사를 읽고 번역.
5. 스페인어권 독자에게 맞게 각색.
6. 이름, 숫자, 인용문을 확인.
7. 최대한 빠르고 정확하게 송고.
8. AP, Reuters 등 다른 대형 뉴스 에이전시와 속도 경쟁.

핵심은 **“빠르게, 하지만 틀리면 안 된다”**입니다. 출처: `epitech part1`, 26:52–28:50

---

## 2) ARTE 웹 퍼블리셔의 일상

참석자 3은 매일 오전 10시에 편집 회의를 열고, 문화·뉴스 등 여러 팀이 모여 어떤 프로그램을 웹사이트와 SNS에서 강조할지 결정한다고 했습니다.

ARTE는 초긴급 속보 전문 채널은 아니므로, 장기 다큐멘터리, 슬로우 인포, 픽션/TV 시리즈 등을 뉴스 상황과 TV 편성에 맞춰 조정합니다.

핵심 업무는 **콘텐츠 선택, 배치, 플랫폼별 강조점 조정**입니다. 출처: `epitech part1`, 28:50–29:45

## 3) 언론 자유 NGO/RSF 계열 업무

참석자 4는 기자가 구금, 납치, 공격을 당한 사례나 언론 자유 침해 문제를 추적합니다. 담당 지역이 넓고, 일부 국가는 군사정권이나 분쟁 지역이라 직접 접근이 어렵습니다.

업무 방식:

출처: `epitech part1`, 29:45–32:01; 38:04–41:15

# 4. 기자들이 말한 핵심 Pain Point

## 1) 속도: breaking news는 몇 분 단위 싸움

참석자 2는 뉴스 에이전시 업무에서 가장 큰 어려움은 **속도**라고 했습니다.

예시:

| 상황 | 요구 시간 |
| --- | --- |
| 긴급 알림/alert | 최대 5분 이내 |
| 기사 작성/번역 | 약 15분 내외 |
| 대형 속보 | 더 빠른 판단 필요 |

지진, 전쟁, 대통령 사망, 국제 분쟁 같은 큰 뉴스는 “정확하지만 매우 빠르게” 처리해야 합니다. 출처: `epitech part1`, 33:25–34:45

---

## 2) 정확성: 뉴스 에이전시는 고객이 100% 신뢰함

참석자 2는 뉴스 에이전시의 고객 언론사들이 받은 내용을 다시 검증하지 않는다고 설명했습니다. 즉, 고객은 에이전시가 보낸 정보가 **100% 정확하다고 전제**합니다.

그래서 오류가 나면 단순 오타가 아니라, 기관 신뢰와 비즈니스 자체에 타격이 됩니다. 출처: `epitech part1`, 34:45–35:24

---

## 3) 번역 AI의 한계: 빠르지만 뉘앙스와 고유명사를 틀림

참석자 2는 DeepL 같은 번역 도구가 속도에는 도움이 되지만, 다음 문제가 있다고 했습니다.

| 문제 | 설명 |
| --- | --- |
| 뉘앙스 손실 | 문장의 미묘한 의미 차이를 놓침 |
| 고유명사 오류 | 이름 표기 방식이 내부 스타일과 다를 수 있음 |
| 문맥 오류 | 특정 문장이 다른 의미로 해석될 수 있음 |
| 스타일 불일치 | 뉴스 에이전시의 표준 표현과 다르게 번역됨 |

그래서 필요한 것은 **AI 번역을 쓰되, 인간 편집자가 검증하는 균형**입니다. 출처: `epitech part1`, 35:24–36:10

---

## 4) 퍼블리셔의 문제: 마지막 검문소 역할

참석자 3은 publisher가 뉴스와 리포트가 나가기 전 **마지막 gatekeeper**라고 설명했습니다.

검증 대상:

| 검증 항목 | 내용 |
| --- | --- |
| 이름 | 인물명 정확성 |
| 장소 | 지명, 국가, 현장 위치 |
| 정보 | 사건 내용이 사실인지 |
| 전달 방식 | 플랫폼에 맞는 문구인지 |
| 신뢰성 | 게시해도 되는 수준인지 |

또한 현대 기자는 한 가지 일만 하는 것이 아니라 촬영, 리포팅, 편집, 배포까지 다 해야 하므로 업무가 점점 더 복합화되고 있다고 했습니다. 출처: `epitech part1`, 36:31–37:51

---

## 5) 전통 미디어 신뢰 하락

참석자 3은 프랑스 등에서 전통 미디어에 대한 신뢰가 낮아지고 있다고 말했습니다. 사람들이 전통 매체를 떠나면 수익도 사라지고, 미디어는 생존 위기에 처할 수 있다고 했습니다.

핵심 문제는 **정확성뿐 아니라 신뢰 회복**입니다. 출처: `epitech part1`, 37:13–37:51

---

## 6) 접근 불가능 지역의 정보 수집

참석자 4의 가장 큰 문제는 **물리적으로 접근할 수 없는 지역에서 정보를 얻는 것**입니다.

예시:

| 상황 | 문제 |
| --- | --- |
| 군사정권 지역 | 외부 기자/NGO 접근 제한 |
| 분쟁 지역 | 안전 문제로 현장 취재 불가 |
| 구금시설 | 사진, 영상, 내부 정보 접근 불가 |
| 현지 제보자 | 외국 NGO와 접촉했다는 이유로 위험해질 수 있음 |

그래서 전직 수감자, 인근 주민, 현지 NGO, 시민사회, 목격자, CCTV, 이동 경로, 동료 증언 등을 종합해 검증한다고 했습니다. 출처: `epitech part1`, 38:04–41:15; `epitech part2`, 22:27–23:18

# 5. AI가 기자 업무에 미치는 영향

## 1) 참석자 2: DeepL, ChatGPT, Copilot, 내부 도구 사용

참석자 2는 1.5~2년 전까지만 해도 AI를 거의 쓰지 않았지만, 속도 압박 때문에 이제는 AI와 “친해질 필요가 있다”고 했습니다.

사용 사례:

| AI 사용 | 설명 |
| --- | --- |
| DeepL | 번역 보조 |
| ChatGPT/Copilot | 긴 기사 요약 |
| ChatGPT | 짧은 헤드라인 아이디어 |
| 내부 번역 도구 | 아직 DeepL보다 부족하다고 평가 |
| 내부 transcription tool | 1시간 인터뷰도 타임스탬프와 함께 빠르게 전사, 매우 유용 |

특히 transcription tool은 긴 인터뷰를 빠르게 처리할 수 있어 업무 시간을 크게 줄여준다고 했습니다. 출처: `epitech part1`, 44:55–49:19

---

## 2) AI 사용의 위험: 없는 내용을 추가할 수 있음

참석자 2는 ChatGPT에게 헤드라인을 부탁하면, 가끔 원문에 없는 내용을 추가할 수 있다고 했습니다. 따라서 AI 결과물은 그대로 쓰면 안 되고, 각 문장과 사실이 실제 기사에 있는지 확인해야 합니다.

즉, AI는 **초안 생성기**이지 **검증자**가 아닙니다. 출처: `epitech part1`, 47:17–48:03

---

## 3) 참석자 3: 문구 작성 보조로 사용

참석자 3은 대통령 사망, 지진 같은 긴급 상황에서 홈페이지나 플랫폼에 올릴 문구를 빠르게 만들기 위해 ChatGPT를 사용할 수 있다고 했습니다.

## 4) 참석자 4: AI는 기자를 대체하지 못함

AI는 전사, 문서 검색, 웹 리서치, 번역에는 도움이 되지만, **취재원의 신뢰 구축과 민감 정보 판단은 인간 기자의 영역**이라고 했습니다. 출처: `epitech part1`, 50:28–54:07

## 3) 다만 “검증할 부분을 하이라이트하는 AI”는 유용할 수 있음

- 이란-이스라엘 같은 대형 속보 상황에서는 정보가 너무 빠르게 쏟아짐.
- 기자는 30분 안에 전체 상황을 설명하는 기사를 써야 함.
- 하지만 새 정보가 계속 들어와서 모든 기사를 훑는 데 시간이 낭비됨.

원하는 도구:

| 기능 | 설명 |
| --- | --- |
| 키워드 입력 | 예: Iran, Israel |
| 관련 기사/속보 수집 | 해당 키워드가 들어간 정보 자동 수집 |
| 짧은 단락 정리 | 사건 흐름을 작은 문단으로 설명 |
| 업데이트 반영 | 새 정보가 들어오면 구조화 |
| 기자 시간 절약 | 기자는 검색보다 기사 작성에 집중 |

이건 해커톤 주제로 매우 강한 pain point입니다. 기자가 직접 “그런 도구가 있었으면 좋았을 것”이라고 말했습니다. 출처: `epitech part2`, 24:01–25:21

# 11. 정보 출처 검증 방식 정리

기자들의 공통 검증 방식은 다음입니다.

## 기자식 검증 공식

1. **출처 확인**
누가 말했는가? 공식 기관인가? 현장 기자인가? 익명 제보자인가?
2. **1차 자료 확인**
공식 발표, 원문 영상, 연설문, 보고서, X/Twitter 원문, 현장 자료를 찾는다.
3. **복수 출처 대조**
한 명 또는 한 매체만 보고 쓰지 않는다.
4. **현장성 확인**
영상/사진이면 언제, 어디서 찍혔는지 본다.
5. **맥락 확인**
제보자가 어떤 agenda를 갖고 있는지 확인한다.
6. **동료와 상의**
의심스러운 경우 혼자 판단하지 않고 팀과 논의한다.
7. **최종 책임은 인간 기자**
AI가 아니라 기자가 최종 판단한다.

출처: `epitech part1`, 18:41–22:59; `epitech part2`, 11:14–15:29; 21:15–23:18

# 14. 해커톤 발표 전략용 기계식 정리

## 문제 정의 공식

1. 기자가 겪는 반복 업무를 찾는다.
2. 그 업무가 왜 시간 압박을 만드는지 말한다.
3. 틀렸을 때 어떤 피해가 나는지 말한다.
4. 현재 AI가 왜 완전 해결하지 못하는지 말한다.
5. 우리가 만든 도구는 AI가 판단하는 것이 아니라 **기자가 판단할 재료를 줄여주는 도구**라고 정의한다.

# 15. 최종 요약

이 녹취록에서 가장 중요한 결론은 다음입니다.

1. 해커톤은 **실제 기자 pain point를 기술 프로토타입으로 해결**하는 것이 목표다.
2. 발표는 **5분 피치 + 작동 데모 + 소스코드 링크 + 시각자료 + AI 사용 설명**이 필요하다.
3. 기자들의 공통 문제는 **속도, 정확성, 신뢰, 검증, 정보 과부하**다.
4. 기자들은 AI를 완전히 신뢰하지 않지만, **번역·요약·전사·검색·검증 후보 표시**에는 유용하다고 본다.
5. 가장 강한 프로젝트 후보는 **속보 키워드 기반 정보 수집/요약 도구** 또는 **기사 오류 후보 하이라이트 도구**다.
6. 발표에서 중요한 포인트는 “AI가 기자를 대체한다”가 아니라 **AI가 기자가 검증할 대상을 줄여준다**는 방향이다.
